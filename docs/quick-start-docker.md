# Quick Start — Docker Deployment

This guide walks a system administrator through running Codex using Docker Compose.

---

## Prerequisites

| Requirement | Notes |
|-------------|-------|
| [Docker](https://docs.docker.com/get-docker/) | 20.10+ |
| [Docker Compose](https://docs.docker.com/compose/install/) | V2 (`docker compose`, not `docker-compose`) |
| Adobe Analytics API credentials | See step 2 |

---

## 1. Get the Code

```bash
git clone https://github.com/aerobeehq/codex.git
cd codex
```

Or download and extract a release archive — the `Dockerfile` and `docker-compose.yml` are included.

---

## 2. Set Up Credentials

Codex reads credentials from a **secrets directory** mounted into the container. Each client gets its own JSON file.

### 2a. Create the secrets directory

```bash
mkdir -p secrets
```

This directory sits **next to** `docker-compose.yml` and is already referenced in the compose file.

### 2b. Create a client config file

```bash
cp config.dist.json secrets/acme.json
```

Name the file after your client or report suite. The filename stem becomes the client slug in URLs (`/acme/`).

### 2c. Fill in your credentials

Open `secrets/acme.json` and replace every placeholder:

```json
{
    "APP_TITLE": "Acme Corp",
    "AW_REPORTSUITE_ID": "acmeprod",
    "API_VERSION": "2.0",
    "CLIENT_ID": "your-client-id",
    "CLIENT_SECRET": "your-client-secret",
    "ORGANIZATION_ID": "ABC123@AdobeOrg",
    "SCOPES": "openid, AdobeID, additional_info.projectedProductContext",
    "AW_USERNAME": "user:acme",
    "AW_SECRET": "wsse-secret"
}
```

See the [Configuration reference in README.md](../README.md#configuration) for a full description of every field.

> **Never commit `secrets/`** to version control. The directory is already listed in `.gitignore`.

---

## 3. Build and Start

```bash
docker compose up -d --build
```

- `--build` rebuilds the image whenever `Dockerfile`, `pyproject.toml`, or application code changes. Omit on subsequent starts if nothing has changed.
- `-d` runs in detached (background) mode.

Expected output:

```
[+] Building ...
[+] Running 1/1
 ✔ Container codex  Started
```

---

## 4. Verify It's Running

```bash
docker compose ps
```

You should see the `codex` container with status `Up`. Then open:

- **Brochure site:** `http://localhost:5010/`
- **Client dashboard:** `http://localhost:5010/acme/` (replace `acme` with your client slug)

---

## 5. View Logs

```bash
# Follow live logs
docker compose logs -f

# Last 100 lines
docker compose logs --tail=100
```

The cache warmer logs show when each client's data has been pre-fetched:

```
[acme] Cache warmed: evars
[acme] Cache warmed: props
...
```

---

## 6. Stop / Restart

```bash
# Stop containers (keeps data volumes)
docker compose down

# Restart without rebuilding
docker compose up -d

# Force rebuild (e.g. after a code update)
docker compose up -d --build
```

## Volume Mounts

The `docker-compose.yml` mounts three directories from the host:

| Host path | Container path | Mode | Purpose |
|-----------|---------------|------|---------|
| `./secrets` | `/app/secrets` | read-only | Per-client credential files |
| `./exports` | `/app/exports` | read-write | CSV exports generated by the app |
| `./cache` | `/app/cache` | read-write | Cached API responses (survives restarts) |

> The `cache/` directory persists between container restarts. This means the background warmer picks up where it left off without a cold-start delay. Clear it with `rm -rf cache/` if you want to force a full refresh.

## Adding a Second Client

Drop another JSON file in the `secrets/` directory and restart:

```bash
cp config.dist.json secrets/betacorp.json
# edit secrets/betacorp.json with different credentials
docker compose restart
```

The new client is available at `http://localhost:5010/betacorp/` immediately after restart.

## Exposing Codex via a Reverse Proxy

For production deployments, run Codex behind Nginx or a similar reverse proxy so you can terminate TLS and use a custom domain.

### Minimal Nginx config

```nginx
server {
    listen 443 ssl;
    server_name codex.example.com;

    ssl_certificate     /etc/ssl/codex.crt;
    ssl_certificate_key /etc/ssl/codex.key;

    location / {
        proxy_pass         http://127.0.0.1:5010;
        proxy_set_header   Host $host;
        proxy_set_header   X-Real-IP $remote_addr;
        proxy_set_header   X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header   X-Forwarded-Proto $scheme;
    }
}
```

No changes to `docker-compose.yml` or `Dockerfile` are needed — Codex listens on `0.0.0.0:5010` inside the container.

## Updating Codex

```bash
git pull origin main
docker compose up -d --build
```

Docker's layer cache means only changed layers are rebuilt — usually just the application code layer.

## Troubleshooting

| Symptom | Likely cause | Fix |
|---------|-------------|-----|
| Container exits immediately | Bad credentials or missing `secrets/` directory | Check `docker compose logs` for the startup error |
| `RuntimeError: CODEX_SECRETS_DIR is not set` | Volume mount missing | Verify `./secrets:/app/secrets:ro` is in `docker-compose.yml` |
| `No valid client configs found` | `secrets/` directory is empty or files have invalid JSON | Validate with `python -m json.tool secrets/acme.json` |
| Port 5010 already in use | Another process on the host | Change the host port: `"5011:5010"` in `docker-compose.yml` |
| Exports not saving | `exports/` not writable | Ensure `./exports:/app/exports:rw` in `docker-compose.yml` |
| API 1.4 pages show an error | Adobe 1.4 API unreachable | Adobe's 1.4 infrastructure is being deprecated; Codex retries automatically |
