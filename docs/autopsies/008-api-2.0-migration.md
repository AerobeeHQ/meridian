# Issue 008: Adobe Analytics API 2.0 Migration Implementation

**Date:** March 13, 2026  
**Type:** Feature / Migration  
**Status:** Completed  
**Branch:** `feature/migrate-to-api2.0`

## Objective

Migrate the Codex application from Adobe Analytics API 1.4 (WSSE authentication) to Adobe Analytics API 2.0 (OAuth2 authentication) following the migration plan documented in `/docs/api-migration.md`.

## Problem Statement

The application was using Adobe Analytics API 1.4 with legacy WSSE authentication. As Adobe phases out the 1.4 API, the application needed to migrate to the 2.0 API which uses modern OAuth2 authentication.

## Solution

### Hybrid API Approach

Not all 1.4 API endpoints have equivalents in 2.0. The solution implements a **hybrid approach**:

| Endpoint | API Version | Reason |
|----------|-------------|--------|
| Props | **2.0** | Available via `/dimensions` endpoint |
| eVars | **2.0** | Available via `/dimensions` endpoint |
| Events | **2.0** | Available via `/metrics` endpoint |
| ListVars | 1.4 | Better config data in 1.4 |
| Processing Rules | 1.4 | **Not available in 2.0** |
| Marketing Channels | 1.4 | Better config data in 1.4 |
| Channel Rules | 1.4 | **Not available in 2.0** |

### Files Created

1. **`app/services/adobe_auth.py`** - OAuth2 authentication module
   - Token acquisition via Adobe IMS
   - Automatic token caching with 5-minute refresh buffer
   - Supports configurable scopes

2. **`app/services/adobe_analytics_v2.py`** - API 2.0 service
   - Discovery endpoint for `globalCompanyId`
   - Dimensions and metrics endpoints
   - Response transformation to match 1.4 format

### Files Modified

1. **`app/__init__.py`** - Load OAuth2 config fields
2. **`app/routes/main.py`** - Config-based API version switching
3. **`config.dist.json`** - Document new OAuth2 fields
4. **`verify_setup.py`** - Check OAuth2 config, recommend `uv sync`

## Configuration

### New `config.json` Fields

```json
{
    "API_VERSION": "2.0",
    "CLIENT_ID": "<from Adobe I/O Console>",
    "CLIENT_SECRET": "<from Adobe I/O Console>",
    "SCOPES": "openid, AdobeID, additional_info.projectedProductContext",
    "ORGANIZATION_ID": "<org>@AdobeOrg"
}
```

### Prerequisites

1. Create OAuth Server-to-Server credential in Adobe I/O Console
2. Add Adobe Analytics API to the project
3. Select appropriate product profile with Analytics permissions
4. Copy credentials to `config.json`

## Technical Details

### OAuth2 Token Flow

```
1. POST https://ims-na1.adobelogin.com/ims/token/v3
   - grant_type: client_credentials
   - client_id, client_secret, scope
   
2. Response: { access_token, expires_in }

3. Token cached for ~24 hours (with 5-min refresh buffer)
```

### API 2.0 Request Headers

```
Authorization: Bearer {access_token}
x-api-key: {client_id}
x-gw-ims-org-id: {org_id}
```

### Response Transformation

API 2.0 returns dimensions/metrics with different structure than 1.4. The service transforms responses to maintain compatibility:

- `variables/prop1` → `prop1`
- `variables/evar1` → `evar1`
- `metrics/event1` → `event1`

## Key Decisions

1. **Direct API implementation** over third-party library (consistency, control)
2. **OAuth2 Server-to-Server** over JWT (simpler, no certificates)
3. **Hybrid 1.4 + 2.0** approach for full functionality
4. **Config-based switching** via `API_VERSION` field

## Lessons Learned

1. API 2.0 doesn't expose all admin config (processing rules, channel rules)
2. Discovery endpoint required to get `globalCompanyId`
3. Response formats differ significantly - transformation layer needed
4. Some endpoints (listVars, channels) have better data in 1.4
